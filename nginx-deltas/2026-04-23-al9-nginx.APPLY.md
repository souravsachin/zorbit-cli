This runbook adds nginx `location` blocks for the 9 ZMB-scaffolded app
modules AY deployed into the `zu-apps-al9` container (ports 3050-3058)
so that `https://zorbit-uat.onezippy.ai/api/<module_slug>/...` resolves
through our `zu-nginx` reverse proxy.

Underscore-slug nomenclature per J73. Do not switch to hyphens.

## 3. Install the delta as a separate include file

Two options. Option A (RECOMMENDED) keeps the delta isolated so it
can be rolled back by deleting one file.

### Option A — separate include (preferred)

Precondition: the zu-nginx server block for `zorbit-uat.onezippy.ai`
already contains an `include /etc/nginx/conf.d/*.conf;` (or
`include /etc/nginx/conf.d/zorbit-uat-*.conf;`) directive OUTSIDE but
referenced from inside the server block. Verify with:

```
docker exec zu-nginx grep -rn 'include' /etc/nginx/conf.d/ || true
grep -n 'include' /home/sourav/apps/zorbit-platform/nginx/conf.d/zorbit-uat.conf
```

If a matching include directive exists inside the server block, drop
the delta next to it:

```
sudo cp /tmp/2026-04-23-al9-nginx.conf \
        /home/sourav/apps/zorbit-platform/nginx/conf.d/zorbit-uat-al9-nginx.conf
ls -la /home/sourav/apps/zorbit-platform/nginx/conf.d/
```

If the server block does NOT have an include directive, fall back to
Option B.

### Option B — append to the live conf

```
cat /tmp/2026-04-23-al9-nginx.conf \
  >> /home/sourav/apps/zorbit-platform/nginx/conf.d/zorbit-uat.conf
```

CRITICAL: the `location` blocks in the delta are designed to live
INSIDE the existing `server { ... }` block for `zorbit-uat.onezippy.ai`.
If Option B appends them AFTER the closing `}` of that server block,
nginx will reject the config. Inspect the last 20 lines before and
after the append:

```
tail -40 /home/sourav/apps/zorbit-platform/nginx/conf.d/zorbit-uat.conf
```

If the delta ended up outside the server block, restore from the
backup made in step 2 and use Option A instead.

## 5. Verify public HTTPS reaches all 9 modules

From any shell with internet access (laptop or sovpn). 9 curls:

```
for slug in \
    network_empanelment_application \
    network_empanelment_workflow \
    hi_claim_initiation \
    hi_claim_adjudication_workflow \
    hi_claim_decisioning \
    hi_customer_portal \
    retail_banking_demo \
    wealth_mgmt_demo \
    manufacturing_demo ; do
  printf "%-45s " "$slug"
  curl -s -o /dev/null -w "%{http_code}\n" \
    "https://zorbit-uat.onezippy.ai/api/$slug/api/v1/G/health"
done
```

Expected: every row ends in `200`.

If any row is `404`, the location block is present but the internal
NestJS route is not at `/api/v1/G/health` — check the container logs
(`docker logs zu-apps-al9 --tail 200 | grep <slug>`).

If any row is `502` or `504`, nginx reached upstream but the service
is not answering on that port inside `zu-apps-al9`.

Also spot-check the manifest endpoint for one module:

```
curl -s https://zorbit-uat.onezippy.ai/api/network_empanelment_application/api/v1/G/manifest | jq '.moduleId, .manifestVersion'
```

Expected: `"zorbit-app-network_empanelment_application"` and `"2.3"`.

## 6. What this delta does NOT do

- Does NOT wire up cross-module CRUD access through nginx. The SDK's
  `ZorbitEntityCrudModule` mounts CRUD under its own `api/<slug>/...`
  prefix inside the container; combined with this delta's rewrite-
  strip, external CRUD URLs would double up the slug. If the unified
  console needs public HTTPS access to these modules' CRUD routes,
  a follow-up delta is required. Tracked as J86 in the session report.
- Does NOT restart `zu-apps-al9`. Services keep running.
- Does NOT update the unified console, sidebar, or module registry.
